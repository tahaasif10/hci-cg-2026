Norman vs. GOMS Comparison: The Wi-Fi Toggle Case

Based on Dix et al., Human–Computer Interaction (Ch. 12), this table analyzes how different models handle a routine task like enabling Wi-Fi on a modern smartphone.

Comparison of Norman’s Action Cycle and GOMS Concepts
Norman’s Stage Task Step (iPhone/Samsung) GOMS Component Alignment Status
Goal (The Intention) "I want to connect to the internet." GOAL: Enable-Wi-Fi ✅ Direct Match
Plan (Action Sequence) Deciding to swipe down for Quick Settings. METHOD: Swipe + Tap ✅ Conceptual Match
Specify (Identify Action) Locating the Wi-Fi icon visually. Not Explicitly Modeled ❌ GOMS assumes knowledge
Execute (Physical Act) Physical finger tap on the screen. OPERATOR: Tap [K] ✅ Direct Match
Perceive & Evaluate Seeing the icon turn blue. Outside GOMS Scope ❌ Evaluation Gulf

Theoretical Insight: Closing the Gulfs

The Gulf of Execution

In our Wi-Fi example, if the user doesn't know they need to swipe down, there is a large Gulf of Execution. GOMS helps measure how long it takes once the user knows the method, but Norman's model helps us identify why the user might get stuck.

The Who Paid Problem

As discussed in class, complex interactions (like splitting a bill) require multiple cycles of Norman's model because the "Goal" changes based on the "Evaluation" of the previous step. GOMS is less effective here because it struggles with non-routine planning.
